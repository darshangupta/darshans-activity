# Darshan's Activity — Design

## Overview

A personal, responsive web app for training toward a half marathon (Aug 16, 2026) and a marathon (Dec 25, 2026), while also logging lifting/boxing. Pulls running data automatically from Strava daily, recommends a running schedule, and gives full editing control over the plan. Single user, no auth — accessed via a Vercel URL.

## Tech Stack

- Next.js 16 (App Router), TypeScript, Tailwind 4, shadcn/ui
- Vercel Postgres for storage
- Vercel Cron for the daily Strava sync job
- No auth — open at the Vercel deployment URL
- Repo: `darshans-activity`, GitHub + Vercel deployment

## Data Model

- **`races`** — `id, name ('half' | 'marathon'), date, created_at`. Seeded with Half (2026-08-16) and Marathon (2026-12-25).
- **`plan_config`** — single-row JSON config: weekday template (which days are run days + base mileage ranges), ramp rate, taper weeks before each race. Edited via Settings; drives plan regeneration.
- **`planned_workouts`** — one row per date: `date (PK), kind ('run' | 'open'), target_min, target_max, is_override, note`. Generated by the plan algorithm. Rows with `is_override = true` are manual edits and are skipped by regeneration.
- **`strava_activities`** — `strava_id (PK), date, distance_mi, moving_time_s, avg_pace, splits (jsonb), synced_at`. Cached daily pull from Strava, matched to `planned_workouts` by date.
- **`strength_logs`** — `id, date, activity_type ('lift' | 'box'), duration_min, note, created_at`. Manual entries, no prescribed schedule.
- **`strava_tokens`** — single-row: `access_token, refresh_token, expires_at`. The web app's own Strava API connection (separate from any Claude/MCP Strava connection — this app talks to Strava's REST API directly).

## Training Plan Algorithm

Plan spans today through the marathon, in four phases:

1. **Half Build** (now → Aug 16, ~7 weeks): long run ramps up weekly toward a peak, then tapers the final 1-2 weeks.
2. **Recovery** (1 week post-half): reduced-mileage reset.
3. **Marathon Build** (~16 weeks): long run resumes ramping from the recovery baseline toward a marathon peak.
4. **Taper** (final ~3 weeks before Dec 25): long run steps back down.

Weekday behavior (from `plan_config`, all editable):

- **Monday** — fixed easy run, default 3 mi.
- **Wednesday** — run within a configurable range (default 3-5 mi), scaling toward the top during peak build weeks and the bottom during recovery/taper.
- **Saturday** — the long run, driven by the phase progression above.
- **Tue/Thu/Fri/Sun** — `kind: 'open'`. No prescribed run; these are the days for logging lifting/boxing.

Generation is a pure function: `(plan_config, races, date_range) → planned_workouts[]`. Re-run whenever config/template changes, but never overwrites `is_override` rows.

Default seed values for `plan_config`: long run starts ~6 mi, ramps +1 mi/week, tapers to 50% two weeks before a race; marathon build resumes from a ~4-5 mi recovery-week long run and ramps to an ~18-20 mi peak. These are starting points, fully adjustable from Settings (ramp rate, peak caps, taper length).

## Strava Sync

- **One-time setup**: register a dedicated Strava API app (strava.com/settings/api) for `client_id`/`client_secret`. One-time OAuth authorize flow via `/api/strava/connect` to obtain the initial refresh token, stored in `strava_tokens`.
- **Daily cron** (`/api/strava/sync`, ~6am via `vercel.json` crons config): refresh the access token if expired, call Strava's `GET /athlete/activities` for activities since the last sync, filter to runs, fetch lap/stream data for mile splits, upsert into `strava_activities` keyed by `strava_id`.
- **Matching**: an activity's date links it to that day's `planned_workouts` row, so the UI shows planned vs. actual (e.g. "planned 5mi, ran 5.3mi @ 8:12/mi").
- **Manual sync**: a "sync now" button for on-demand refresh outside the daily cron.

This is a separate Strava connection from any MCP-based Claude integration — the app authenticates with Strava's REST API directly, server-side.

## Calendar UI & Editing

- **Home screen**: month view (shadcn/ui-styled grid). Each day cell color-coded — run (planned distance, or actual once synced), open/lift-box day (shows strength log if present), rest.
- **Day detail panel** (click a day): planned workout, editable inline (changing it sets `is_override = true`); Strava data if synced (splits table); strength-log form if it's an open day.
- **Settings page**: edit the weekly template (which days run, base ranges) and algorithm knobs (ramp rate, peak caps, taper length). Saving regenerates all future non-overridden `planned_workouts`.

## Strength Logging

From a day's detail panel: type toggle (lift/box), duration in minutes, free-text note, submit → `strength_logs`. No prescribed schedule — log whenever you actually did something.

## Testing

- Unit tests for the plan generator (pure function): correct phase transitions, taper behavior, override preservation.
- Unit tests for Strava sync matching logic (mocked API responses).
- Manual in-browser testing for the calendar UI and mobile responsiveness — personal tool, no heavy E2E investment.
